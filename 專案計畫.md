# 花蓮單車碳減量兌換網站 — 專案計畫

確認需求：14 個站點採類 YouBike 共享單車（人力車＋電輔車，實體智能鎖），騎乘起訖以車鎖回報事件為準、輔以 GPS 即時定位算里程，技術棧為 Next.js + Supabase，獎品兌換需要真實庫存與序號管理。以下計畫依此規劃。

## 0. 關於「連結真的碳權」的策略定位（先讀這段）

查證後確認：台灣「溫室氣體自願減量專案管理辦法」規定能申請註冊減量專案、取得可交易額度的資格只有「事業」與「中央/地方政府機關」，個人開發的網站不是合格申請人，也沒有公開機制能讓一個 App 直接核發受政府承認的碳權。但地方政府機關（也就是花蓮縣政府本身）是合格申請人之一。

因此目前定案的策略是：**先把網站做成一本「可信、防作弊、資料經得起檢視」的減碳量存摺**，對外文案用「減碳量 / 環保積分」，不對外宣稱這是正式碳權。同時把系統架構設計成未來如果花蓮縣政府想正式申請自願減量專案（自行車／運具移轉類，是否已有可套用的核准方法學需要之後再查證環境部溫室氣體自願減量暨抵換資訊平臺 https://carbonoffset.moenv.gov.tw/ 最新公告清單），這套系統可以直接作為監測（MRV）資料的基礎建設。這個角色定位（幫縣政府解決執行面痛點，而不是自己發不受承認的碳權）比較容易換來縣政府的重視，之後有意願再往正式申請那條路推進，那條路本身涉及找查證機構、方法學適用性確認，時程通常數月到一年以上，且最終決定權在縣政府而非你的網站。

**補充（重要）**：你提供的路線圖是花蓮縣政府「山海一線 低碳騎旅」計畫的實際文件，總經費 3,500 萬元的基礎工程案（改善路面、標誌標線、沿途驛站），不是虛構的路線。這代表你的網站不是憑空想像一個計畫去跟縣政府提案，而是可以直接對接一個縣政府已經編列預算在做的實際建設案，未來去談合作時可以直接說「你們已經在蓋硬體了，我這邊把數位監測/使用者端的軟體先做出來」，說服力比空手去提案高很多。

**再補充（關鍵升級，來自「兩潭兩鐵及白鮑溪」計畫文件）**：這份文件揭露了兩個對 Ecomiles 非常有利的事實，之前的策略要跟著升級：

1. 花蓮縣政府自己就規劃要做「個人碳帳戶（淨零花蓮 APP）」，並且明講要「參考基北北桃減碳存摺概念，加入騎乘兩潭、兩鐵及白鮑溪自行車道的減碳行動」。這代表 Ecomiles 的減碳存摺功能不是你自己發明一個新故事去說服縣政府，而是可以直接定位成「這個縣政府已經規劃要做的個人碳帳戶構想的雛型／示範模組」，說服力等級完全不同。
2. 資金路徑比之前講的「自願減量專案」更具體：中央「溫室氣體管理基金」已編列 17.1 億元專款補助地方政府溫室氣體減量工作，縣市政府只要每年 10 月前提交「溫室氣體減量執行方案」，經審核通過就能拿到補助，不用全部自籌。Ecomiles 產出的騎乘減碳數據（防作弊、可追溯的那套設計）正好可以作為這份年度方案的佐證資料/監測工具，時程比等「自願減量專案方法學核准」快得多也具體得多。

因此「政府儀表板」的文案建議調整方向：主打「可作為溫室氣體減量執行方案佐證資料，協助爭取碳費基金補助」，而不是只講「理論碳權市場價值」；市場行情換算可以保留當作附加資訊，但不是主要賣點了。

其他可以在簡報口頭講、不用寫進程式的信任狀：花蓮縣溫室氣體排放量已從 2005 年 1,179 萬噸降至 2023 年 541 萬噸，減幅 54%，超過聯合國建議的 42% 減量目標；已完成 26 處示範基地建設；願景是效法丹麥、挪威等國際低碳典範城市。這些數字建議在簡報開場當作「為什麼這件事現在重要」的背景，不用做成app裡的功能。

## 1. 需不需要 Perplexity 深度研究

需要，但只針對幾個「查錯會出大問題」的點做，不必做泛用型研究：

1. **花蓮縣站點與路網官方資料**：確認你手上的 14 站座標是否與花蓮縣政府 / [data.gov.tw](https://data.gov.tw) 上的官方開放資料一致（站名、經緯度、路線里程）。若之後站點異動，網站資料才有正式來源可對照，也能放在網站上做為資料來源引用。
2. **「碳權」用詞的法遵風險（已查證）**：在台灣，「碳權」是有法律定義的名詞，需經第三方查證（MRV 原則）並在政府登錄後才能交易，目前政府尚未開放個人碳權交易機制。查證時發現巨大集團的「YouBike 碳權存摺」確實對外用了「碳權」二字做行銷，但那本質上是企業自建積點系統、不是真的可交易碳權，屬於灰色地帶的行銷用詞。你的專案規模較小、非上市公司行銷案，建議保守處理：對外一律用「減碳量」「碳積分」「環保點數」，不使用「碳權」，避免使用者誤以為能拿去交易或抵稅。這點已經寫進 CLAUDE.md 的命名規則，不用再另外查證。
3. **騎乘減碳量的換算係數（已查證，見下方說明）**：已找到可用的數字，寫進 CLAUDE.md 了。依環保署機動運具空氣汙染物排放係數，四行程機車（目前主流機車）約 0.0508 kgCO2e/km，電動機車約 0.0252、小客車約 0.208、電動小客車約 0.078（出處：https://www.tcsif.org/news_detail/TCSIF-NEW11，原始數據源自張宇琦、柏雲昌 2021 年研究，查證日期 2026-07-06）。建議固定用「取代四行程機車」這一組係數，公式為 `減碳量(kg) = 里程(km) × 0.0508`，對外文案註明計算假設即可，不用同時列多種假設造成混亂。網路上流傳「騎車每公里減碳 65 公克」的說法，是巨大集團高層受訪時引用的機車+汽車混合估算值、不是環保署正式係數，不建議直接採用。上線前仍建議到環境部氣候變遷署平台核對是否有更新版本的官方係數表。

   **「一個人能省下多少碳權」怎麼算**：單次騎乘的減碳量 = 里程 × 上述係數，例如騎 5 公里 ≈ 0.25 公斤 CO2e（以取代機車估算）。若想做「全站/全縣累積減碳量」的展示（類似 YouBike 官方每年公布全區減碳量的做法），只要把所有已完成 `rides` 的 `carbon_saved_kg` 加總即可，可以放在首頁或做一個公開頁面向花蓮縣政府/大眾展示整體成效，這是可以列入之後版本的加分功能，不影響 MVP 開發順序。
4. **GPS 里程計算與地圖匹配（map-matching）作法**：GPS 直接算的距離會因訊號漂移而失真，值得查一下業界常見的簡化處理方式（例如 Haversine 累積距離 + 移動速度過濾雜訊點），不需要做到專業導航等級的路網貼合。
5. **獎品序號兌換的消費者權益眉角**：若獎品涉及儲值、電子禮券或高單價實體商品，稍微查一下台灣對「線上發放兌換序號」是否有基本的消保揭露義務（多半是提醒你要寫清楚兌換期限、售完為止等條款），非強制但能省事。

這五項比較適合「先查完再開工」，其他架構面的問題（怎麼寫 Supabase schema、怎麼設計等級制度）不需要外部研究，直接在下面的計畫裡定案即可。

## 2. 系統架構總覽

- **前端 / 後端**：Next.js（App Router），一套專案同時處理前端頁面與 API Route，部署到 Vercel。
- **資料庫 / 驗證 / 即時定位儲存**：Supabase（PostgreSQL + Auth + Realtime + Storage）。
- **地圖**：Mapbox GL JS 或 Google Maps JavaScript API 顯示站點與騎乘軌跡（擇一，Mapbox 免費額度較寬鬆）。
- **GPS 追蹤**：瀏覽器 `navigator.geolocation.watchPosition`，騎乘中每隔幾秒記錄一個座標點，結束時用 Haversine 公式累加距離。
- **狀態管理**：Zustand 或 React Context 皆可，畫面不複雜用 Context 就夠。

## X. 群眾外包地理標記資料（Flickr 照片 CSV）延伸應用

隊友提供的花蓮路線 bounding box 內 Flickr 地理標記照片資料集（5,854 筆，2017~2026，含經緯度、拍攝時間、作者、標籤），除了原本規劃的熱力圖／人流引導／AI 智慧問答，已進一步分析並定案以下三項功能，資料已預先算好放在 `data/` 目錄（`photo-popularity.json`、`photo-monthly-pattern.json`、`photo-yearly-trend.json`、`hidden-hotspots.json`），避免前端直接載入 1.3MB 的原始 CSV。

### X.1 人氣加權路線規劃（完整邏輯）

背景限制：這條路線本質上是**線性**的（14 站固定地理順序，海線走到底接吉安山線），不是任意兩點互通的路網，所以「路線規劃」不是通用的最短路徑問題，而是「在使用者的里程/時間預算內，從目前所在站往一個方向，挑一段區間，讓這段區間的加權人氣總分最大化」的區間選擇問題，14 個節點直接窮舉所有 `O(n²)` 種起訖組合即可，不需要複雜演算法。

**單站分數 Score(i)** = 由四個項目組成：

1. **人氣基底 PopularityNorm(i)**：取 `photo-popularity.json` 的 `photoCount500m`，做 `log(1+count) / log(1+全站最大值)` 正規化到 0~1（用 log 是因為七星潭 713 張跟花蓮港景觀橋 3 張差距太大，線性正規化會讓其他站分數趨近於 0）。
2. **當令加成 SeasonalRelevance(i, m)**：取 `photo-monthly-pattern.json` 該站 `monthlyCounts[m] / max(monthlyCounts)`，代表「現在這個月份，對這一站來說是不是相對熱門的時機」，0~1。
3. **政策加權 PolicyBonus(i)**：`segment == "jian"` 給 ×1.2，`coastal` 給 ×1.0——這是把「把人流導向吉安段」的政策目標直接寫進演算法權重，不是只在文案上喊口號。
4. **擁擠懲罰 CrowdingPenalty(i, m)**：`max(0, monthlyCounts[m] / 該站12個月平均 - 1.5) × 懲罰係數`，如果這個月這一站的人氣是自己全年平均的1.5倍以上（代表本來就旺季爆棚），扣一點分數，讓演算法不要一直把人往最擁擠的地方送，順便呼應「過度觀光管理」的角色。

`Score(i) = (0.4 × PopularityNorm(i) + 0.3 × SeasonalRelevance(i,m)) × PolicyBonus(i) − 0.2 × CrowdingPenalty(i,m)`

（權重 0.4/0.3/0.2 是起始值，之後可以依實際測試手感微調，不用一次定死。）

**區間分數與路線挑選**：先用 Mapbox Directions API 把 14 站兩兩相鄰的實際騎乘距離量一次、存成靜態陣列（不要用直線距離，山海線有彎折）。使用者目前最近站為 `c`、里程預算為 `D` 公里時，窮舉所有從 `c` 出發、總距離 ≤ `D` 的連續站點區間 `[start, end]`，計算：

`IntervalScore = Σ Score(i)，i ∈ [start, end]  −  λ × 該區間總距離`

`λ` 是小係數，避免演算法無腦選最長的區間。分數最高的 1~2 個區間即為推薦路線。如果 `hidden-hotspots.json` 裡有隱藏熱點落在區間內兩站之間、且偏移距離在 400 公尺內，額外加一點小加成（其自身正規化人氣的一半），並在生成文字時提示「順路可以繞去 XXX」。

這個分數計算完全在後端用純數學算好，AI（DeepSeek）只負責把算好的區間**轉成友善的中文推薦文字**，不負責決定路線本身——這樣政府儀表板要秀「為什麼推薦這條路線」時，可以直接攤開算式和數字，而不是一句「AI 覺得不錯」，可信度高很多。

### X.2 單日路線序列分析（已初步分析，結論見下）

方法：把每筆照片依「作者＋拍攝日期」分組，組內依時間排序，每張照片對應到最近的官方站點（超過 800 公尺不強行對應，避免失真），去除連續重複站點後就是這個人當天的移動序列。

**目前分析結果**：5,854 筆裡有效組成「作者+日期」338 組，其中只有 48 組出現跨站移動（其餘 290 組整天都只在同一站附近拍照，代表大部分是單點停留而非移動路徑）。這 48 組裡最常見的相鄰站點轉移**全部集中在海線北段**（例如「農好基地⇄七星潭」10+9次、「朝金定置漁場⇄七星潭」6+6次、「花蓮酒廠⇄奇萊鼻燈塔」5+4次），沒有任何一筆觀察到跨到吉安/南段的移動紀錄。

**我的建議**：48 筆樣本量太小、且完全來自路線工程完工前的舊照片，不建議直接拿來當人氣加權演算法裡的「主要」輸入（容易對 48 筆雜訊資料過度解讀）。比較務實的用法分三層：

1. **現在（黑客松）**：當成簡報敘事素材，不寫進演算法——「歷史資料顯示北段景點之間本來就有人潮自然串聯，但完全沒觀察到有人把行程延伸到吉安段，這正是這條路線工程要補上的斷點，也正是 Ecomiles 遊戲化機制要解決的問題」，比單純的人氣數字更有故事性。
2. **上線後（有真實 rides 資料時）**：`rides` 表本身就會累積起訖站資料，等有一定筆數後，用同一套「相鄰站點轉移」邏輯重跑一次，這時候樣本是你們自己使用者的真實騎乘行為、樣本量會遠大於 48 筆，那時候才適合正式納入 X.1 演算法，例如給「歷史上常被串聯造訪」的站點組合一個小加成。
3. **如果現在就想要有點用**：可以只當一個很小的加分項（例如有觀察到歷史轉移紀錄的站點對 +5% 分數），而不是主要權重，避免過度依賴小樣本。

### X.3 打卡密度指數與韌性敘事

**擁擠指數**：`CrowdingIndex(站, 月) = 該站該月人氣 / 該站12個月平均人氣`，>1 代表這個月比這一站平時更擁擠。可以在政府儀表板直接秀「本月擁擠指數：七星潭 1.6×（明顯高於平時）」這種一句話統計，比單純的人氣排行更貼近「過度觀光管理」的實際決策用途（學術上這類分析常用於國家公園承載量評估）。

**韌性敘事（意外發現）**：`photo-yearly-trend.json` 的年度總數 2023 年（1259 張）到 2024 年（483 張，減少 61.6%）到 2025 年（789 張，回升 63.4%），時間點剛好對上 2024 年 4 月花蓮大地震。這不是嚴謹的因果證明（同時期也可能有其他因素），但作為政府儀表板或簡報裡的敘事亮點很合適：「這條路線撐過了地震衝擊，現在正在回穩」，建議在儀表板的年度趨勢圖旁邊加一行文字說明，並註明資料來源是 Flickr 公開地理標記照片、非官方觀光統計，避免被誤認為正式統計數據。

**資料代表性但書（建議一併放進儀表板頁尾）**：Flickr 使用者在年齡、國籍、攝影愛好者比例上都有偏差，不完全等於實際遊客分布，這批資料適合當作「趨勢方向」的參考證據，不建議當成精確的遊客人數統計來引用。
